## 2024-05-24 - Mass Assignment Vulnerability in User Registration
**Vulnerability:** The public `POST /api/users` endpoint allowed users to specify their `role`, `department`, and `skills` in the request body during registration.
**Learning:** This is a classic Mass Assignment vulnerability. It existed because the backend implicitly trusted `req.body` to define user properties, relying only on defaults (`role: role || 'customer'`) rather than strictly filtering or hardcoding values. This allowed any public user to elevate their privileges by sending `{"role": "admin"}`.
**Prevention:** Always hardcode or strictly whitelist (e.g., using a schema validation library or explicitly mapping fields) sensitive fields like `role`, `permissions`, `isAdmin`, etc., during resource creation or update. Do not pass `req.body` directly or rely on truthy checks that can be overridden by user input.
## 2025-02-12 - Prevent NoSQL and Regex Injection in Express/Mongoose controllers
**Vulnerability:** The `getTicketList` endpoint blindly passed `req.query` parameters (`status`, `priority`, etc.) directly to the MongoDB `find()` filter, allowing for NoSQL injection (e.g. `?status[$ne]=Closed`). Additionally, the `search` query parameter was passed unescaped to a `$regex` operator, risking ReDoS and Regex injection vulnerabilities.
**Learning:** `req.query` parameters in Express can be parsed as arrays or objects (e.g. `req.query.status[$ne]=Closed` becomes an object `{ $ne: "Closed" }`), which Mongoose natively executes. Directly interpolating unescaped user input into Regex queries without boundaries permits malicious matching patterns.
**Prevention:** Always explicitly enforce parameter types before query execution (`typeof req.query.param === 'string'`) to defend against object-based NoSQL injections, and escape dynamic regex inputs using an established pattern (`search.replace(/[.*+?^${}()|[\]\\]/g, '\\$&')`).
